**Ocena stacku vs PRD**

- MVP szybkość: Angular 21 + Material wspiera szybkie UI (siatka, modale, formularze), lecz Spring Boot 4 + JPA +
  Postgres to większy narzut niż lżejszy backend (np. NestJS + Postgres); nadal wykonalne.
- Skalowalność: Architektura (Angular SPA, Spring Boot + Postgres) skaluje się, ale pojedynczy VPS ogranicza
  dostępność/obciążenie; brak cache/kolejek/logów strukturalnych trzeba dodać przy wzroście.
- Koszt utrzymania: Dwa duże runtime’y zwiększają koszt devops/infra (buildy, RAM JVM). Na małym VPS może być ciasno
  przy serwowaniu frontu i backu równocześnie.
- Złożoność: Jak na MVP to stack „enterprise-heavy” – osobne buildy/CI, większa powierzchnia konfiguracji.
- Prostszą alternatywą: zostawić Angular (lub Vite + Angular dla lżejszego dev), uprościć backend do NestJS/Express +
  Prisma/TypeORM + Postgres; front jako statyczne pliki + mały Node serwer za reverse proxy na VPS.
- Bezpieczeństwo: Spring Security daje mocną bazę (hash haseł, JWT/sesje, CSRF/CORS, rate limiting) przy poprawnej
  konfiguracji; w alternatywie Node trzeba to świadomie zbudować. RODO wymaga obsługi usunięcia danych i eksportu
  niezależnie od stacku.

**Rekomendacja**

Stack spełni wymagania, ale jest ciężki dla MVP na jednym VPS. Jeśli zespół czuje Springa – zostań, planując od razu:
konfigurację auth (hash, JWT/refresh), migracje DB, monitoring/logowanie, rate limiting. Jeśli priorytet to szybkość i
lekki footprint, rozważ lżejszy backend (NestJS/Express) przy zachowaniu Postgresa.
